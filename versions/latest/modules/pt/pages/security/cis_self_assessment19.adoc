== Visão Geral

Este documento é um complemento ao guia de proteção do RKE2. O guia de proteção fornece orientações prescritivas para a proteção de uma instalação de produção do RKE2, e este guia de benchmark é destinado a ajudá-lo a avaliar o nível de segurança do cluster protegido em relação a cada controle no benchmark do CIS Kubernetes. Deve ser utilizado por operadores de RKE2, equipes de segurança, auditores e tomadores de decisão.

Este guia é específico para a linha de lançamento *v1.27-1.32* do RKE2 e a versão *v1.9* do Benchmark do CIS Kubernetes.

Para mais informações sobre cada controle, incluindo justificativas detalhadas e descrições de verificações, você pode consultar a seção correspondente do Benchmark do CIS Kubernetes v1.8. Você pode baixar o benchmark, após criar uma conta gratuita, no https://www.cisecurity.org/benchmark/kubernetes/[Centro para Segurança da Internet (CIS)].

=== Metodologia de testes de controles

Cada controle no Benchmark do CIS Kubernetes foi avaliado em um cluster RKE2 configurado de acordo com o guia de proteção correspondente.

Estes são os resultados possíveis para cada controle:

* *APROVADO* - O controle é automatizado (pontuação: verdadeiro). O cluster RKE2 em teste passou na auditoria descrita no benchmark.
* *Não Aplicável* - O controle não é aplicável (tipo: pular) ao RKE2 devido à forma como foi projetado para operar. A seção de justificativa explicará por que isso ocorre.
* *AVISO* - O controle é manual (pontuação: falso) no benchmark do CIS e depende da intervenção manual do operador. A seção de remediação fornecerá orientações sobre como alcançar um resultado APROVADO.

==== 3.2.2 Certifique-se de que a política de auditoria aborde as principais preocupações de segurança (Manual)

**Resultado:** AVISO

**Remediação:** Revise a política de auditoria fornecida para o cluster e certifique-se de que ela aborde pelo menos as seguintes áreas: - Acesso a Segredos gerenciados pelo cluster. Deve-se ter cuidado para registrar apenas Metadados para solicitações a Segredos, ConfigMaps e TokenReviews, a fim de evitar o risco de registrar dados sensíveis. - Modificação de objetos Pod e Deployment. - Uso de `pods/exec`, `pods/portforward`, `pods/proxy` e `services/proxy`. Para a maioria das solicitações, é recomendado registrar minimamente no nível de Metadados (o nível mais básico de registro).
